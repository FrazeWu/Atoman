# 发现与决策

## 需求
音乐模块不是单纯上传和播放系统，而是一个以**专辑为中心条目**的维基式音乐资料库。用户可以自由上传、编辑、替换资源、回看历史，并通过讨论、回滚、保护机制处理争议。

## 核心产品结论

### 1. 专辑是基础对象
所有修改都以专辑为基础，而不是以单首歌为基础。

这意味着：
- 专辑是详情页主体
- 专辑是编辑页主体
- 专辑是历史页主体
- 专辑是争议和保护的主体
- 歌曲只是专辑内部结构的一部分

### 2. 单曲也是专辑
用户上传一首歌时，不单独创建“孤立歌曲条目”。
而是：
- 创建一个 `type = single` 的专辑
- 该专辑只包含一首歌
- 后续所有编辑都围绕这个单曲专辑进行

这让系统模型统一为：
- single = 1 首歌的专辑
- EP = 少量歌曲的专辑
- album = 常规专辑

### 3. 维基模式的核心不是“谁上传谁做主”
不同用户之间出现意见不同，不能用上传者最终决定的逻辑。

维基模式应采用：
1. 允许直接修改
2. 所有修改保留历史
3. 通过编辑说明和讨论说明理由
4. 若出现编辑战，由管理员或受信任用户回滚和保护条目

## 维基如何解决不同用户意见不同

### 第一层：先允许修改
维基的基本逻辑不是“先审批再改”，而是“先改，事后纠正”。
所以当两个用户对：
- 封面
- 年份
- 曲序
- 艺术家归属
- 歌词版本
- 音频版本
有不同意见时，系统先允许他们提交不同版本。

### 第二层：保留完整历史
每次修改都进入 revision 历史。
这样冲突不是“覆盖旧答案”，而是“产生多个可审视版本”。

### 第三层：靠理由和来源形成共识
用户需要通过：
- 编辑说明
- 讨论页
- 来源引用
来解释自己为什么修改。

谁的内容更接近稳定版本，不看谁先上传，而看：
- 是否有来源
- 是否更准确
- 是否更完整
- 是否得到社区共识

### 第四层：编辑战时介入治理
如果两个用户来回改同一条目，就形成编辑战。
这时采用维基常见做法：
- 回滚明显破坏版本
- 临时保护条目
- 要求到讨论页达成共识
- 管理员保留一个临时稳定版

## 推荐的治理结构

### 条目状态
- `open`：开放编辑
- `disputed`：存在明显争议
- `protected_semi`：半保护，普通用户不能直接生效
- `protected_full`：全保护，仅管理员可编辑
- `hidden`：内容暂时隐藏

### 用户分层
1. 游客：浏览、播放、查看历史
2. 登录用户：创建、编辑、替换资源、写编辑说明、参与讨论
3. 管理员/受信任用户：回滚、保护、解除保护、裁定争议、隐藏恶意内容

## 编辑粒度结论
一次“专辑编辑”可能修改：
- 专辑标题
- 艺术家
- 发行日期
- 封面
- 专辑简介
- 曲目顺序
- 单首歌标题
- 歌词
- 音频文件

这些不应被拆成多个分散的治理中心，而应归为一次 **Album revision**。

## 版本机制结论
- 每次编辑生成一个 Album revision
- revision 快照应包含：
  - 专辑元数据
  - 曲目列表
  - 每首歌的标题 / 曲序 / 歌词 / 音频引用 / 封面引用
- 回滚不是恢复旧记录，而是创建一个新的 revert revision

## 创建逻辑结论
创建入口应叫“创建专辑”或“制作专辑”，不是“上传歌曲”。

原因：
- 这更符合统一模型
- 用户即便只有一首歌，也是在创建单曲专辑
- 后续所有历史、讨论、保护都能围绕同一个条目展开

## 首期 MVP 结论
第一版建议只打通：
1. 创建专辑
2. 单曲自动成为 single 类型专辑
3. 编辑专辑及其歌曲结构
4. 记录 Album revision
5. 查看专辑历史
6. 回滚专辑版本
7. 条目保护
8. 编辑说明
9. 争议时依靠讨论 + 回滚 + 保护处理

## 当前最重要的设计原则
- **专辑高于歌曲**：歌曲是专辑内部内容，而不是首期主条目。
- **先改后协商**：维基模式优先允许编辑，而不是预审。
- **历史是第一治理工具**：所有争议都必须能追溯。
- **讨论几轮就能锁定**：音乐条目事实空间有限，不需要复杂仲裁体系。
- **两层权限足够**：普通用户 + 管理员，不需要中间信任层级。
- **统一模型优先于局部方便**：single / EP / album 都走一套逻辑。

## 运行规则简化结论（2026-05-09）

### 核心判断
音乐条目的事实空间比维基百科的政治/历史文章小得多：
- 专辑名、发行年份、曲目列表、艺术家——这些信息有明确的对错
- 不需要多轮仲裁，几个回合讨论就能锁定

因此，维基的复杂信任层级和升级路径**不需要照搬**：
- 不需要自动确认/扩展确认/回滚者/审核者
- 不需要 RfC / 调解 / 仲裁委员会
- 只需要两层权限 + 三种状态

### 三种状态
- **Open**：开放编辑，任何登录用户可直接修改
- **Disputed**：有争议讨论中，继续开放编辑但标注争议提示
- **Confirmed**：已锁定的稳定版本，需管理员解锁才能再次编辑

### 两层权限
- **普通登录用户**：编辑 Open / Disputed 条目，参与讨论，对 Confirmed 条目提议修改
- **管理员**：裁定争议，确认/锁定条目，解锁条目，回滚恶意修改

### 音频文件结论
- 普通用户可以替换音频
- 旧音频保留在历史，随时可回滚
- 不需要独立审核队列
- 若有争议，管理员从历史回滚即可

## 下一步最适合继续细化的内容
- 讨论页首期是否上线，还是先只靠编辑说明承担协商功能
- Confirmed 状态下”提议修改”的具体交互流程
- 争议状态是否在列表页显示
- 专辑创建页、编辑页、历史页的完整用户路径
