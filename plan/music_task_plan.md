# 任务计划：音乐模块维基化开发规划

## 目标
将音乐模块从“用户上传与播放”扩展为“用户自由上传、编辑、协作维护、可回溯修订历史”的维基式音乐平台，并以**专辑为中心条目**统一承载单曲、EP、专辑等音乐内容。

## 当前阶段
Phase A-F 全部完成（2026/05/09）。音乐模块维基化实现完毕。

## 已做决策
| 决策 | 理由 |
|------|------|
| 音乐模块采用”维基式协作编辑”而非仅作者中心模型 | 目标是构建可持续完善的公共音乐资料库 |
| 音乐模块以专辑为基础对象 | 专辑天然承载封面、年份、艺术家、曲序、歌曲列表等结构信息 |
| 单曲视作”只有一首歌的专辑” | 统一模型，避免 album / single / song 走两套逻辑 |
| 默认允许登录用户编辑他人上传的资料 | 维基模式要求开放协作编辑 |
| 普通用户可以替换音频文件 | 用户已确认开放该能力，但需配合历史与治理机制 |
| 所有编辑都必须写入 revision 历史 | 维基模式的前提是可追溯与可恢复 |
| 回滚通过”基于旧版本创建新版本”实现 | 避免破坏历史链条，审计更清晰 |
| 不同用户意见冲突按维基模式解决 | 不靠”上传者说了算”，而靠历史、讨论、保护、回滚解决 |
| 删除应弱化，优先使用保护、隐藏、回滚 | 维基系统更强调可恢复而非硬删除 |
| 首期 revision 使用 snapshot_json | 实现更简单，回滚和审计更直接 |
| 首期创建入口以”创建专辑”而非”创建歌曲”为主 | 让用户从一开始就进入统一条目模型 |
| 条目状态分三种：Open / Confirmed / Disputed | 覆盖全部场景，Disputed 可循环回 Confirmed |
| 权限只分两层：普通用户 / 管理员 | 音乐资料事实空间有限，多层信任体系过重 |
| 讨论页里话题为普通讨论，无 PR 机制 | 纯维基模式，管理员负责裁定，不靠用户提交 PR 触发流程 |
| Confirmed/Disputed 由管理员手动控制 | 管理员判断条目稳定性，不依赖自动触发 |
| 管理员职责三件事：确认、重新开放、解决冲突 | 职责清晰，不参与日常编辑 |
| 管理员入口在音乐模块导航，仅管理员可见 | 和普通用户界面共存，不单独建后台 |
| 管理员入口页显示 Open 待确认 + Disputed 有争议两类条目 | PR 去掉后管理员核心工作变为监控条目状态 |
| 创建专辑时内联新建艺术家（方案 B） | 不打断创建流程，体验最流畅 |
| 内联新建艺术家表单：名字 + 头像 | 足够轻量，详细信息后续在 Artist 页面补充 |
| 创建专辑页：单页滚动 | 字段不多，分步反而增加操作成本 |
| 专辑类型自动建议：1首 Single / 2-6首 EP / 7首+ Album | 符合业界惯例，用户可手动覆盖 |
| Disputed 状态在列表页显示争议标记 | 让用户感知哪些条目有争议，保持透明 |
| 创建流程：先创建艺术家，再创建专辑 | 专辑必须关联艺术家，保证数据结构完整 |
| Artist 条目走完整维基化，与 Album 同一套流程 | 统一模型，所有实体行为一致 |
| Artist 成员列表：纯文本，不关联 Artist 条目 | 首期保持简单，避免过度设计 |
| Artist 分类字段首期跳过 | 等风格分类体系建好后再加，避免早期数据混乱 |
| Artist 支持多艺名：主名 + 别名列表 | 同一人可能有多个名字，别名参与搜索索引 |
| 重复条目合并：用户讨论页 @管理员，管理员执行合并 | 普通用户无合并权限，防止误操作 |
| 合并时主条目由管理员手动指定，另一个永久重定向 | 保留 URL 连续性，不破坏已有链接 |
| 合并时被合并条目主名自动成为主条目别名 | 保留历史名字，搜索可达 |
| 合并时 revision 历史、专辑、讨论页全部迁移并按时间合并 | 完整保留两个条目的历史 |

## 运行规则 v1（专辑为核心）

### 核心对象
- **Album**：主条目，所有编辑、历史、争议、回滚都围绕专辑发生
- **Song**：专辑中的曲目，不作为独立治理中心；其变更归入所属专辑 revision
- **Artist**：关联对象，可后续独立维基化，但首期从属专辑编辑场景

### 单曲规则
- 平台不存在“脱离专辑的单曲条目”
- 单曲发布时，系统自动或显式创建一个 `type = single` 的专辑
- 该专辑只包含一首 Song
- 后续所有资料修改仍然在这个“单曲专辑”条目下完成

### 编辑粒度
用户可以编辑的内容包括：
- 专辑标题
- 艺术家
- 发行日期
- 封面
- 专辑简介
- 歌曲列表
- 歌曲标题
- 曲序
- 歌词
- 音频文件

这些修改统一视为“对专辑条目的一次修订”，而不是分散的 song patch。

### 版本规则
- 每次保存都生成一条 **Album revision**
- revision 快照包含：专辑元数据 + 曲目列表 + 曲目元数据 + 资源引用
- 回滚到旧版本时，生成新的 revision，并把该快照重新设为当前状态

## 运行规则 v2（简化版，专辑为核心）

### 设计原则
音乐条目的事实空间相对固定，不像维基百科的政治/历史条目那样复杂。
因此：
- 不需要多层信任体系
- 不需要复杂仲裁路径
- 只需要：创建 → 编辑 → 确认 → 有异议再开放

### 完整生命周期

```
创建艺术家
    ↓
创建专辑（归属该艺术家）
    ↓
Open（开放编辑，持续修改完善）
    ↓ 管理员或发起人确认资料稳定
Confirmed（锁定，稳定版本）
    ↓ 有人提交修改建议或提出异议
Disputed（重新开放，讨论中）
    ↓ 讨论几轮达成共识
Confirmed（再次锁定）
```

Disputed 之后可以反复循环回 Confirmed，不是一次性流程。

### 条目状态说明

**Open（开放编辑）**
- 新创建的条目默认进入此状态
- 任何登录用户可直接编辑
- 每次修改必须填写编辑说明
- 所有版本进历史
- 列表页不显示特殊标记

**Confirmed（锁定确认）**
- 管理员确认当前版本为稳定资料后进入
- 普通用户不能直接编辑
- 可提交”修改建议”（类 PR）：写明想改什么、为什么
- 管理员审核建议，结合讨论页意见决定是否合并
- 合并后生成新 revision，条目继续保持 Confirmed
- 列表页不显示特殊标记

**Disputed（有争议）**
- 有用户对 Confirmed 版本提出异议且未解决时进入
- 条目重新开放编辑（回到类 Open 状态）
- 讨论页开放，双方陈述理由
- 列表页显示争议标记，用户可感知
- 管理员裁定后重新 Confirmed

### 用户权限（两层）

**普通登录用户**
- 创建艺术家
- 创建专辑
- 编辑 Open / Disputed 状态的条目
- 对 Confirmed 条目提交修改建议
- 在讨论页发言

**管理员**
- 将条目从 Open 推进到 Confirmed
- 将 Confirmed 条目回退到 Disputed
- 审核并合并修改建议
- 回滚恶意修改
- 隐藏/删除内容

### 创建逻辑

**创建艺术家**
- 独立流程，Artist 是条目，可后续完善
- 创建后即 Open，可随时编辑

**创建专辑**
- 必须关联已有艺术家
- 必填：专辑名、艺术家、至少一首歌
- 可选：封面、发行日期、简介、歌词
- 若只上传一首歌，`type = single`
- 创建完成即 Open，自动生成初始 revision

**修改建议流程（Confirmed 条目）**
- 用户提交建议，写明修改内容和理由
- 建议进入待审列表
- 管理员在讨论页和建议中看完整上下文
- 管理员合并 → 生成新 revision，继续 Confirmed
- 管理员拒绝 → 建议关闭，但讨论页记录保留

### 音频文件处理
- 普通用户可上传新版本音频（Open / Disputed 状态）
- 旧音频保留在 revision 历史
- Confirmed 状态下音频替换走修改建议流程
- 有争议时管理员从历史回滚

### 讨论页
- 每个专辑、艺术家条目都有独立讨论页
- 用于说明修改理由、协商争议
- 讨论页记录永久保留，是条目历史的一部分
- 首期直接上线，不推迟


### 创建逻辑
- 用户进入“创建专辑”流程
- 必填：专辑名、艺术家、至少一首歌
- 可选：封面、发行日期、简介、歌词、附加曲目
- 若用户只上传一首歌，系统仍创建 single 类型专辑
- 创建完成后立即生成初始 revision

### 页面逻辑
- 详情页中心对象是专辑
- 编辑页中心对象是专辑
- 历史页中心对象是专辑
- 歌曲只作为专辑内部结构显示

## 首期 MVP 范围
### 必做

- Album revision 详情

## 验收标准
- 创建单曲时，系统创建一个 single 类型专辑
- 编辑任意歌曲字段，本质上生成一条 album revision
- 专辑历史页能看到完整修订时间线
- 回滚后不会覆盖旧历史，而是新增 revert revision
- 争议时可以通过历史 + 说明 + 保护机制维持稳定版本
- 用户从前端感知到“音乐条目是可协作维护的专辑页”，而不是孤立歌曲页
