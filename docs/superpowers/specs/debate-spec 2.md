# Debate Spec

## Purpose
辩论模块用于承载结构化公共论证，而不是普通聊天讨论。它必须支持议题发起、多立场展开、论点树组织、证据引用、观点表态与阶段性总结沉淀。

## Core Model

### Core objects
- `DebateTopic`
- `DebatePosition`
- `DebateArgument`
- `DebateEvidence`
- `Vote`
- `Conclusion`
- `Moderation`

### Structural rules
- 辩论模块采用结构化论点树，而不是扁平评论流。
- 一个议题可以包含多个立场，不强制仅分正反。
- 证据应独立建模，以便追踪来源和复用。
- 投票只表达立场，不直接决定结论真伪。
- 总结与结论必须允许迭代更新。

## Product Goals
- 支持高质量表达与结构化论证。
- 让证据与论点形成清晰关系。
- 让读者可以回看讨论演化与结论沉淀。
- 与社区模块明确分工：社区偏日常交流，辩论偏结构化论证。

## Page Structure

### Debate home
- 议题列表
- 分类与排序
- 发起议题入口

### Topic detail
- 议题信息
- 多立场结构
- 论点树
- 投票/表态区
- 总结区
- 证据与引用入口

### Argument tree
- 父子论点关系
- 对立/支持关系
- 引用证据挂接

### Summary and timeline
- 阶段性总结
- 观点与证据演化回顾

## Governance Rules
- 议题、论点、证据都需要基础治理能力。
- 管理员可折叠、置顶、锁定与处理争议内容。
- 证据引用必须可追溯。
- 辩论系统需要质量工具，但不应退化为普通论坛互动模式。

## Durable Decisions
- 论点树是模块的核心，不可退化为简单评论流。
- 多立场比二元正反更符合现实议题复杂性。
- 证据独立建模是模块价值的一部分，而不是装饰能力。
- 结论是迭代生成物，不是一次性静态摘要。

## Boundaries
- 当前阶段仍处于产品与范围规划完成状态，不代表业务代码已完整落地。
- 是否允许匿名浏览但限制发言仍可后续决策。
- 是否做证据可信度标注、拖拽重组、裁判/主持人角色，可作为增强层处理。
