# Timeline Spec

## Purpose
时间线模块用于承载历史事件整理、关系关联、可视化浏览与知识沉淀，而不是普通文章列表。它应该把事件、人物、地点、标签与来源组织成一个可检索、可浏览、可追溯的档案型系统。

## Core Model

### Core entities
- `TimelineEvent`
- `TimelinePerson`
- `TimelinePlace`
- `Relation`
- `Tag`
- `Source`

### Event model
- 事件是所有浏览与关联的中心实体。
- 事件至少需要标题、摘要、描述、时间、分类、重要度、状态等字段。
- 时间表达必须支持精确日、月、年、区间与不确定时间。

### Person and place model
- 人物与地点都应独立建模。
- 事件可以关联多个相关人物、地点、其他事件、标签与来源。
- 来源必须可见，以保证档案系统的可追溯性。

## Product Goals
- 让用户录入并维护历史事件。
- 让事件与人物、地点、标签、来源形成知识关联。
- 让读者可以按时间、主题和地理维度浏览资料。
- 让系统具备档案感、长期可扩展性和来源可见性。

## Page Structure
- 时间线主画布
- 时间缩放
- 分类/标签筛选
- 搜索栏
- 事件预览卡片
- 事件详情页
- 人物页 / 地点页

## Durable Decisions
- 时间线以 Event 为核心，而不是帖子或文章。
- 时间表达必须支持不完整和不确定日期。
- 人物、地点、来源必须独立建模，不能只做附属字符串字段。
- 资料来源可见是档案型系统的核心要求。
- 时间线视图需要支持缩放与筛选，以应对不同时间跨度下的信息密度。

## Governance and Quality Rules
- 事件来源需要引用规范。
- 编辑与审核策略需要考虑资料冲突与质量控制。
- 删除、隐藏、合并与可信度标注都属于长期治理层能力。

## Open Boundaries
- BCE 支持、地图联动、用户草稿提交、因果关系类型仍属于后续扩展方向。
